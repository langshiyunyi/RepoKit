# RepoKit 使用说明

RepoKit 是一个在越狱 iPhone 本机运行的越狱源搭建、配置、管理和发布工具。它包含一个 UIKit 图形界面 App 和一个命令行工具 `repokit-helper`：App 负责展示和交互，helper 负责真实文件操作、deb 扫描、索引生成、源检查和 GitHub 推送。

RepoKit 以 rootless 为默认目标，并兼容 roothide。教程里的路径均使用逻辑路径，用户不需要、也不应该手动填写实际 preboot 路径。

## 一、RepoKit 是什么

RepoKit 适合在自有或授权越狱设备上维护个人/团队越狱源，主要能力包括：

- 创建新的本地越狱源
- 导入已有源目录
- 导入单个 deb 文件
- 从已安装软件包重建 deb 并加入源
- 编辑 deb 内部 `DEBIAN/control` 字段和图标
- 生成 `Packages`、`Packages.gz`、`Release`
- 检查重复包、缺失字段、架构不匹配和 deb 文件丢失
- 配置 GitHub remote/branch/user/email/token
- 发布到 GitHub Pages，供 Sileo、Zebra 等包管理器添加

## 二、程序逻辑

RepoKit 的工作流是：

1. 用户在 App 中创建、导入或选择一个源。
2. App 调用 `/usr/bin/repokit-helper` 执行真实操作。
3. helper 读取或写入当前源的 `repo.json`。
4. 导入的 deb 被复制到当前源的 `public/debs/`。
5. helper 使用 `dpkg-scanpackages` 扫描 `public/debs/` 并生成 `Packages`。
6. helper 生成压缩索引，例如 `Packages.gz`，如果工具可用也会生成 `Packages.zst`、`Packages.bz2`、`Packages.xz`。
7. helper 写入 `Release` 源描述文件。
8. helper 从 `Packages` 反向生成 `packages.json`，供 App 快速显示软件包列表。
9. 发布时 helper 在当前源的 `public/` 目录执行 Git 命令。
10. GitHub Pages 生效后，包管理器通过 `Base URL` 读取源索引。

`Packages` 是包管理器读取的权威索引；`packages.json` 只是 RepoKit App 的 UI 缓存。包数量显示异常时，优先重建索引，而不是手动编辑缓存文件。

## 三、越狱环境兼容

### 逻辑路径

RepoKit 对用户暴露逻辑路径：

```text
/var/mobile/RepoKit
/usr/bin/repokit-helper
```

App 和 helper 内部使用 `jbroot()` 适配 rootless/rootHide 的实际路径，因此不要在表单、教程或源配置里填写实际 preboot 路径。

### rootless 与 roothide

默认 rootless 构建：

```sh
make clean && make package
```

roothide 构建：

```sh
make clean THEOS_PACKAGE_SCHEME=roothide && make package THEOS_PACKAGE_SCHEME=roothide
```

rootless 包架构为 `iphoneos-arm64`，roothide 包架构为 `iphoneos-arm64e`。App 和 helper 仍按 `ARCHS=arm64` 编译，这是正常的。

### 外部依赖

核心依赖来自 `control`：

```text
dpkg
dpkg-dev
gzip
zstd
git
```

其中：

- `dpkg` / `dpkg-dev`：读取、解包、重打包 deb，并提供 `dpkg-scanpackages`
- `gzip`：生成 `Packages.gz`，这是核心压缩索引
- `zstd`：生成 `Packages.zst`
- `git`：发布到 GitHub
- `bzip2`、`xz`：如果系统存在，会额外生成 `Packages.bz2`、`Packages.xz`；缺失不影响主流程

### 导入路径建议

导入 deb、图标或已有源时，建议文件放在 mobile 用户可读的位置，例如：

```text
/var/mobile/Documents/my-repo
/var/mobile/test.deb
/var/mobile/icon.png
```

如果导入失败，先检查文件是否真实存在、mobile 用户是否可读、路径是否误用了实际 preboot 路径。

## 四、项目结构

源码项目结构如下：

```text
RepoKit/
├── Makefile
├── control
├── RepoKitApp/
│   ├── Makefile
│   ├── Info.plist
│   ├── Sources/
│   └── Resources/
├── repokit-helper/
│   ├── Makefile
│   └── Sources/
├── layout/
│   └── DEBIAN/postinst
├── repos/
├── packages/
├── logs/
└── 使用说明.md
```

说明：

- `Makefile`：Theos 聚合构建入口，聚合 `repokit-helper` 和 `RepoKitApp`
- `control`：Debian 包元数据，包括包名、描述、作者和依赖
- `RepoKitApp/`：UIKit 图形界面 App
- `RepoKitApp/Sources/`：App Objective-C 源码
- `RepoKitApp/Resources/`：中英文 `Localizable.strings` 等资源
- `repokit-helper/`：命令行 helper，执行核心文件和 Git 操作
- `layout/`：打包安装阶段需要放入系统的文件，例如 `postinst`
- `repos/`：本地源数据，属于运行数据
- `packages/`：Theos 生成的 `.deb` 构建产物
- `logs/`：运行和发布日志
- `.theos/`：Theos 构建缓存，属于生成目录

开源提交时建议不要提交 `.theos/`、`packages/`、`logs/`、`repos/` 等运行或构建产物。

## 五、越狱源结构

RepoKit 的核心数据目录是：

```text
/var/mobile/RepoKit/repos/<repo-id>/
```

每个源目录包含：

```text
/var/mobile/RepoKit/repos/<repo-id>/
├── repo.json
├── packages.json
├── public/
│   ├── CydiaIcon.png
│   ├── Packages
│   ├── Packages.gz
│   ├── Packages.zst
│   ├── Packages.bz2
│   ├── Packages.xz
│   ├── Release
│   ├── debs/
│   └── icons/
├── logs/
└── trash/
```

说明：

- `repo.json`：源配置，包含名称、描述、作者、Base URL、架构、GitHub 设置
- `packages.json`：从 `Packages` 反向生成的 UI 缓存，不是索引权威来源
- `public/`：真正发布给 GitHub Pages 和包管理器读取的目录
- `public/debs/`：实际 deb 文件
- `public/icons/`：软件包图标文件
- `public/CydiaIcon.png`：源图标
- `public/Packages`：越狱源主索引
- `public/Packages.gz`：常用压缩索引，Sileo/Zebra 通常读取它
- `public/Release`：源描述文件
- `logs/`：GitHub 推送等日志
- `trash/`：删除源或文件时使用的回收目录

导入已有源时，RepoKit 会记录该源的发布目录并读取已有索引。例如导入：

```text
/var/mobile/Documents/my-repo
```

该目录会作为当前源的 `publicPath` 管理，后续构建、检查和推送默认操作这个发布目录。

## 七、第一次使用流程

### 1. 安装并刷新图标

安装 deb 后执行：

```sh
uicache -a
```

然后从桌面打开 RepoKit。

### 2. 创建新源

进入底部 `源` 页面，点击创建按钮，填写：

```text
名称：大飞源
描述：个人越狱源
作者：DaFei
Base URL：https://user.github.io/my-repo
Scheme：rootless
Architecture：iphoneos-arm64
```

保存后，RepoKit 会自动创建源目录结构。接着进入 `软件包` 页面导入 deb，再构建索引。

### 3. 导入已有源

进入底部 `源` 页面，点击导入按钮，填写：

```text
路径：/var/mobile/Documents/my-repo
Repo ID：my-repo
名称：大飞源
Base URL：可先留空
```

保存后，RepoKit 会读取已有 `Packages`。如果没有索引，则用 `dpkg-scanpackages` 扫描 `debs/` 并生成 `Packages`、`Packages.gz`、`Release`。

### 4. 如果包数量显示 0

进入底部 `软件包` 页面，点击：

```text
重新扫描 Deb
```

它会重新读取当前源的 `public/debs/`，用 `dpkg-scanpackages` 重建 `Packages`，并把结果缓存到 `packages.json` 供 App 显示。

## 八、如何管理源

- 在源列表中点击一个源，即可切换为当前源。
- 当前源决定软件包列表、构建、检查和 GitHub 推送的目标。
- 在 `源 > 编辑源` 可修改名称、描述、作者、Base URL、模式、架构和源图标。
- 删除源会进入回收目录，方便回滚。
- 导入已有源时，RepoKit 会读取已有发布目录和索引。
- 如果包列表不对，先执行重建索引。

## 九、软件包增删改查

进入 `软件包` 页面后，点击任意软件包，会弹出操作菜单。

### 导入 Deb

导入 Deb 会把选择的 `.deb` 复制到当前源的 `public/debs/`，不会移动或删除原文件。随后 RepoKit 会重建索引。

### 导入已安装软件包

`导入已安装软件包` 会根据 `dpkg` 的已安装记录重建 deb，复制到当前源，然后统一重建索引。适合备份自有设备上已安装的软件包。

### 查看软件包 JSON

软件包列表来自 `Packages` 索引。点击软件包可以查看解析后的 JSON 缓存，字段以 deb control 和 `dpkg-scanpackages` 输出为准。

### 编辑 Control 和图标

编辑会先用 `dpkg-deb -R` 解包，按真实 `DEBIAN/control` 字段填充表单并写回修改；可选复制本地图标到 `public/icons/`，文件名来自 `Name` 字段，并写入 `Icon` 字段，然后用 `dpkg-deb -b` 重打包并重建索引。编辑时原 deb 会移动到 `trash/`。

### 删除 deb 文件

删除操作会把对应 deb 文件移动到 `trash/`，然后重新生成 `Packages`、压缩索引和 `Release`。只删除 `packages.json` 缓存没有意义，因为下一次扫描会重新生成。

## 十、构建和检查源

### 构建索引

点击 `构建索引` 后会生成：

```text
public/Packages
public/Packages.gz
public/Packages.zst（如果 zstd 可用）
public/Packages.bz2（如果 bzip2 可用）
public/Packages.xz（如果 xz 可用）
public/Release
```

### 检查源

点击 `检查源` 后会检查：

- 是否有重复包版本
- 是否缺少 `Package`
- 是否缺少 `Version`
- 架构是否不匹配
- deb 文件是否丢失

输出会显示在弹窗和 `发布` 页面底部的最近输出中。

## 十一、GitHub 发布流程

### 1. 配置 GitHub

进入 `发布` 页面，点击 `GitHub 设置`，填写：

```text
Remote：https://github.com/用户名/仓库.git
Branch：main
Username：GitHub 用户名
Email：Git 提交邮箱
Token：GitHub personal access token
```

RepoKit 会把 Username 和 Email 写入该源 `public/` 目录下的本地 Git 配置，避免提交时出现 `Author identity unknown`。Token 只用于推送，不会写入日志。

### 2. 创建 GitHub 仓库

1. 打开 GitHub，点击右上角 `+`，选择 `New repository`。
2. `Repository name` 建议和源 ID 一致，例如 `my-repo`。
3. 选择 `Public`，可以不勾选模板初始化文件。
4. 创建后复制 HTTPS 地址，例如 `https://github.com/user/my-repo.git`，填到 RepoKit 的 `Remote`。

### 3. 推送 GitHub

点击 `推送 GitHub`，输入提交信息，例如：

```text
Update RepoKit repository
```

RepoKit 会在当前源的 `public/` 目录执行：

```sh
git init
git checkout -B main
git config user.name <username>
git config user.email <email>
git remote add origin <remote>
git add .
git commit -m "Update RepoKit repository"
git push -u origin main
```

日常改完 deb 后可以直接点 `推送 GitHub`。RepoKit 会在推送前自动重新构建 `Packages*`、`Release` 和 `packages.json`；如果构建失败，会停止推送并显示错误。

### 4. 开启 GitHub Pages

第一次 `推送 GitHub` 成功后，在 GitHub 仓库页面进入 `Settings > Pages`：

```text
Source：Deploy from a branch
Branch：main
Folder：/root
```

保存后等待 GitHub Pages 部署完成。源的 `Base URL` 通常填写：

```text
https://user.github.io/my-repo
```

### 5. 添加到 Sileo/Zebra

GitHub Pages 可访问后，在 Sileo 或 Zebra 添加 `Base URL`，例如：

```text
https://user.github.io/my-repo
```

刷新源后应能看到 `Packages.gz` 中的软件包。

## 十二、命令行用法

RepoKit 也可以直接通过命令行管理。

### 导入已有源

```sh
repokit-helper import-source /var/mobile/Documents/my-repo --repo-id my-repo --name "大飞源"
```

### 重新扫描 deb

```sh
repokit-helper rescan my-repo
```

### 构建索引

```sh
repokit-helper build my-repo
```

### 检查源

```sh
repokit-helper check my-repo
```

### 查看软件包列表

```sh
repokit-helper list my-repo
```

### 导入单个 deb

```sh
repokit-helper add my-repo /var/mobile/test.deb --section Tweaks
```

### 编辑 deb control 和图标

```sh
repokit-helper edit-deb my-repo com.example.tweak 1.0 --name "My Tweak" --description "New description" --icon /var/mobile/icon.png
```

### 配置 GitHub

```sh
repokit-helper github my-repo --remote https://github.com/user/repo.git --branch main --user github-user --email you@example.com --token ghp_xxx
```

### 推送 GitHub

```sh
repokit-helper push my-repo --message "Update repo"
```

## 十三、发布前兼容检查清单

发布正式版前建议检查：

- rootless 构建通过：`make clean && make package`
- roothide 构建通过：`make clean THEOS_PACKAGE_SCHEME=roothide && make package THEOS_PACKAGE_SCHEME=roothide`
- App 和 helper 仍为 `ARCHS=arm64`
- rootless deb 为 `iphoneos-arm64`
- roothide deb 为 `iphoneos-arm64e`
- 教程和 UI 中不出现实际 preboot 路径
- 用户路径统一使用 `/var/mobile/RepoKit`、`/usr/bin/repokit-helper`
- `Packages`、`Packages.gz`、`Release` 能生成
- `Base URL` 能访问 `Packages.gz`
- Sileo/Zebra 能刷新源
- GitHub token 不写入 README、日志或提交历史
- `.theos/`、`packages/`、`logs/`、`repos/` 不提交到开源仓库

## 十四、常见问题

### 1. 创建目录失败，没有权限

如果看到类似：

```text
You don't have permission to save the file "debs"
```

说明数据目录权限不对。新版 deb 已带 `postinst` 自动修复。如果仍然失败，可手动执行：

```sh
chown -R mobile:mobile /var/jb/var/mobile/RepoKit
chmod -R u+rwX,go+rX /var/jb/var/mobile/RepoKit
```

### 2. 导入后软件包数量为 0

进入 `软件包` 页面，点击 `重新扫描 Deb` 或 `构建索引`。

也可以命令行执行：

```sh
repokit-helper rescan my-repo
```

### 3. 桌面没有图标或图标没刷新

执行：

```sh
uicache -a
```

必要时重启用户空间。

### 4. GitHub 推送失败

检查：

- `git` 是否安装
- remote 是否正确
- branch 是否正确
- username/email 是否已设置
- token 是否有推送权限
- 网络是否可访问 GitHub
- GitHub Pages 分支是否选择 `main` 和 `/root`

### 5. 软件包字段从哪里来

软件包字段来自 deb 内部 control，并由 `dpkg-scanpackages` 写入 `Packages`。如果要长期修改名称、描述、依赖或图标，请使用“编辑 Control 和图标”重打包 deb。

```

新增 deb 推荐流程：

```
